# Release Runbook

This runbook defines a deterministic release process for EAP.

## Preconditions

- `main` is green in CI.
- Target changes are merged.
- Release draft notes are generated by Release Drafter.
- PyPI Trusted Publisher is configured for stable tag publishing.

## Semantic Version + Changelog Drafting

EAP uses GitHub Release Drafter to automate version resolution and changelog drafting.

Version resolution is label-based:

- `major` -> major bump
- `minor` -> minor bump
- `patch` -> patch bump
- default -> patch

The draft release body is the source of truth for release notes/changelog content.

## Release Steps

1. Ensure PR labels (`major`/`minor`/`patch`) are correct for merged changes.
2. Open the draft release in GitHub (`Releases` page).
3. Verify resolved version and notes.
4. Publish the release with tag `vX.Y.Z`.
5. Confirm release workflow success (`.github/workflows/release.yml`).
6. Verify published package(s) and attach release link in roadmap/issue tracking.

## Publish Authentication

- Stable tags (`vX.Y.Z`) publish via PyPI Trusted Publishing (OIDC, no long-lived token).
- Pre-release tags (`vX.Y.Z-rcN`) use `TEST_PYPI_API_TOKEN` when present.

## Rollback Procedure

Use rollback only when a bad release is published.

1. Stop downstream rollout and communicate issue.
2. If package publication occurred:
   - publish a corrective follow-up release (`vX.Y.(Z+1)`) with fix
   - avoid force-overwriting historical release artifacts
3. If GitHub release/tag must be withdrawn:
   - delete the GitHub release entry
   - delete the bad tag locally and on origin
4. Document incident summary:
   - impact
   - root cause
   - corrective actions

## Post-Release Checklist

- CI release workflow succeeded.
- Release notes are accurate.
- Security scanning workflows remain green.
- Roadmap issue/task status updated.

## V1 Readiness Handoff

- Fully populated `Breaking Changes` and `Upgrade Notes` sections from `docs/release_notes_template.md`

## Security Workflow Notes

- `pip-audit` runs in CI and is always enforced on Python 3.11.
- Optional GitHub Dependency Review workflow can be enabled by setting repository variable:
  - `ENABLE_DEPENDENCY_REVIEW=true`
- GitHub Dependency Graph should be enabled for dependency-review support.
